# HireSense AI — Resume Ranking & Job Match System

Upload a résumé and a job description; get back a match score, missing
skills, recommended keywords, resume improvement suggestions, and a job
fit category — backed by Sentence-Transformer/BERT embeddings, a FAISS
similarity index, and Gemini-assisted suggestions.

```
Résumé (PDF/DOCX/TXT) ─┐
                        ├─► Text extraction ─► BERT/Sentence-Transformer embeddings ─┐
Job description (text)─┘                                                            │
                                                                                      ▼
                              Skill-keyword gap analysis  ◄──────────────  Cosine similarity
                                          │                                         │
                                          └──────────────► Blended match score ◄────┘
                                                                  │
                                                                  ▼
                                                   Gemini (or rule-based fallback)
                                                     resume improvement suggestions
```

## Features

- **Match score (0–100)** blending semantic similarity (Sentence-Transformer
  embeddings) and explicit skill-keyword coverage.
- **Missing skills** and **matched skills**, extracted against a curated,
  extensible skills taxonomy (`app/models/skills_taxonomy.py`).
- **Recommended keywords** to add to the résumé, prioritized by what the
  job needs and the résumé doesn't already say.
- **Resume improvement suggestions** generated by Gemini, with a
  deterministic rule-based fallback so the app works fully offline /
  without an API key.
- **Job fit category** (`Excellent / Strong / Moderate / Weak / Poor Fit`).
- **FAISS-backed similarity search** over a bundled sample résumé corpus
  (real, anonymized résumés from the public Resume Dataset), independent
  of job matching — demonstrates the semantic search layer on its own.
- Upload **PDF, DOCX, or TXT**, or paste résumé text directly.
- **Production hardening**: request-ID tracing through every log line and
  error response, baseline security headers (CSP, X-Frame-Options, etc.),
  per-route rate limiting (a stricter limit on the expensive `/match`
  endpoint than the global default), structured JSON logging, typed
  exceptions mapped to HTTP status codes, CORS allow-list, fail-fast
  config validation in production mode, health/readiness/version probes,
  gunicorn with a dedicated config file, Docker, and a full pytest suite.

## Tech stack

| Layer | Technology |
|---|---|
| Embeddings | `sentence-transformers` (`all-MiniLM-L6-v2` by default — swappable) |
| Vector search | `faiss-cpu` |
| PDF parsing | `pdfplumber` (primary) + `PyPDF2` (fallback) |
| DOCX parsing | `python-docx` |
| Suggestions | Gemini API (`google-generativeai`), with offline fallback |
| Web framework | Flask 3, gunicorn (`gthread` workers, config in `gunicorn.conf.py`) |
| Frontend | Vanilla HTML/CSS/JS (no build step) |
| Tests | pytest, pytest-cov |

## Quickstart (local)

```bash
git clone <this-repo> && cd hiresense-ai
python -m venv .venv && source .venv/bin/activate   # or .venv\Scripts\activate on Windows

make install                  # pip install -r requirements.txt
# A .env file is already included with sane local defaults.
# Open it and paste a real key into GEMINI_API_KEY if you want LLM-generated
# suggestions — https://aistudio.google.com/app/apikey
# Leave it as the placeholder and the app runs fully offline with
# rule-based suggestions instead; nothing breaks either way.

make build-index              # builds the FAISS index from data/sample_resumes.csv
python app.py                 # starts the dev server at http://localhost:5000 (prints a startup banner)
```

Open `http://localhost:5000` in a browser.

## Quickstart (Docker)

```bash
cp .env.example .env          # edit GEMINI_API_KEY etc. as desired
make docker-up                # builds the image (which builds the index) and starts the container
```

The app is then available at `http://localhost:5000`. Health checks:

```bash
curl http://localhost:5000/api/health   # liveness
curl http://localhost:5000/api/ready    # readiness (model + index loaded)
```

## API reference

### `POST /api/match`

Form-data or JSON. Provide **either** `resume_file` (multipart upload) **or**
`resume_text`, plus `job_text`.

```bash
curl -X POST http://localhost:5000/api/match \
  -F "resume_file=@resume.pdf" \
  -F "job_text=We need a backend engineer skilled in Python, Flask, and Docker..."
```

Response:

```json
{
  "match_score": 78.4,
  "semantic_similarity": 71.2,
  "skill_coverage_pct": 83.3,
  "job_fit_category": "Strong Fit",
  "matched_skills": ["python", "flask", "docker"],
  "missing_skills": ["kubernetes"],
  "recommended_keywords": ["kubernetes", "ci/cd", "..."],
  "suggestions": {
    "summary": "Strong alignment, with a few targeted gaps worth closing.",
    "improvement_suggestions": ["..."],
    "rewrite_example": "...",
    "source": "gemini"
  },
  "resume_char_count": 2143,
  "job_char_count": 612
}
```

### `POST /api/extract-text`

Upload `resume_file`; returns the plain text HireSense extracted from it
(useful for a "here's what we read" preview before scoring).

### `POST /api/similar-resumes`

```json
{"text": "Registered nurse with ICU experience...", "top_k": 5}
```

Searches the bundled FAISS index and returns the most semantically
similar résumés (category + snippet + similarity score).

### `GET /api/health` / `GET /api/ready` / `GET /api/version`

Liveness, readiness, and build-metadata probes for load balancers and
orchestrators:

```bash
curl http://localhost:5000/api/health    # {"status": "ok"}
curl http://localhost:5000/api/ready     # {"status": "ready", "checks": {...}}
curl http://localhost:5000/api/version   # {"name": "HireSense AI", "version": "1.0.0", "gemini_enabled": false, ...}
```

`gemini_enabled` reflects whether a real (non-placeholder) `GEMINI_API_KEY`
is configured — useful for confirming which suggestion mode is active
without reading logs.

## Production hardening

- **Request ID tracing** — every response carries an `X-Request-ID`
  header (generated, or echoed back if the caller/proxy already set
  one). The same ID is attached to every structured log line emitted
  while handling that request and to every JSON error body, so a
  single failing request can be traced end-to-end across logs.
- **Security headers** — every response sets `X-Content-Type-Options`,
  `X-Frame-Options`, `Referrer-Policy`, and a `Content-Security-Policy`;
  `Strict-Transport-Security` is added automatically outside debug mode.
- **Per-route rate limiting** — `/api/match` (the expensive endpoint —
  it runs an embedding model and optionally an external LLM call) has
  its own stricter limit (`RATE_LIMIT_MATCH`) on top of the global
  default (`RATE_LIMIT_DEFAULT`) applied to all other `/api/*` routes.
- **Fail-loud config checks** — in production mode (`FLASK_ENV=production`,
  `DEBUG=False`), `Config.validate_for_production()` runs at startup and
  logs a warning for any insecure default (e.g. an unset `SECRET_KEY`)
  so misconfiguration is visible in the logs rather than silent.
- **CORS allow-list** — defaults to `*` for local development, but
  `CORS_ALLOWED_ORIGINS` (comma-separated) restricts `/api/*` to known
  origins in production.
- **gunicorn config** (`gunicorn.conf.py`) — `gthread` workers sized for
  CPU-bound embedding inference rather than the I/O-bound default,
  `preload_app` so the embedding model loads once before forking,
  periodic worker restarts (`max_requests`) to bound memory growth, and
  structured access/error logs to stdout for container log drivers.

## Configuration

All settings are environment variables (see `.env.example`). Key ones:

| Variable | Purpose |
|---|---|
| `GEMINI_API_KEY` | Enables LLM-generated suggestions. Leave as the placeholder (or blank) to use the rule-based fallback. |
| `EMBEDDING_MODEL` | Any `sentence-transformers` model name. |
| `RESUME_CORPUS_PATH` | CSV used by `scripts/build_index.py` to build the FAISS index. |
| `MAX_UPLOAD_SIZE_MB` | Upload size limit (default 10MB). |
| `RATE_LIMIT_DEFAULT` | Rate limit applied to all `/api/*` routes by default. |
| `RATE_LIMIT_MATCH` | Stricter, dedicated rate limit for `/api/match` specifically. |
| `CORS_ALLOWED_ORIGINS` | Comma-separated origin allow-list for `/api/*`. Defaults to `*`. |
| `SECRET_KEY` | Flask session signing key. **Must** be changed before deploying. |
| `APP_VERSION` | Reported by `/api/version`; set from your CI build tag if desired. |

## Using the full résumé corpus

The repo ships a small `sample_resumes.csv` (120 résumés, 5 per category)
to keep the image lean. To index the full corpus (2,484 résumés) instead:

```bash
python scripts/build_index.py --corpus data/full_resumes.csv
```

## Notes on the matching algorithm

The match score blends two independent signals:

- **Semantic similarity (60% weight)** — cosine similarity between
  L2-normalized Sentence-Transformer embeddings of the résumé and job
  text, rescaled from its typical real-world range onto an intuitive
  0–100 scale.
- **Skill-keyword coverage (40% weight)** — the fraction of taxonomy
  skills mentioned in the job description that are also found in the
  résumé, via phrase-boundary (not substring) matching.

This blend is more robust than either signal alone: pure embedding
similarity can be fooled by topically related but under-qualified text,
while pure keyword matching misses synonyms, rephrasing, and context.
